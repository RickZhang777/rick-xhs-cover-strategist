# Rick XHS Cover Strategist GitHub 发布打包说明

这份文档用于把 `rick-xhs-cover-strategist` 发布到 GitHub 和小红书时作为整理依据。Kimi、Claude、Codex 或其他 Agent 可以参考本文件生成 README、发布文案、仓库说明和使用教程。

## 项目定位

`rick-xhs-cover-strategist` 是一个小红书封面策略与提示词生成工具。

它不是单纯的生图 prompt 模板，而是一个轻量对话式工作流：

- 收集个人形象图
- 收集业务画像
- 收集配套素材
- 判断封面重点
- 选择封面视觉风格
- 生成标题方案
- 生成小红书封面生图提示词
- 沉淀可复用的封面规范
- 支持同一聊天内连续生成多张封面
- 支持在“全部重新开始”后继续沿用上一张业务画像或上一张风格
- 支持用户多选时进入 A/B 测试模式

目标用户：

- 小红书博主
- 知识 IP
- 课程/咨询/服务型账号
- 产品、工具、SaaS、AI 工具测评账号
- 需要批量制作封面提示词的内容创作者

## 核心工作流

第一次生成封面时，助手会按三轮收集信息。

### 第一轮：个人形象图

用户上传至多 5 张个人形象图。

提示用户：

```text
请上传至多 5 张个人形象图。

尽量选择脸部轮廓清晰、能识别出人物的照片；正脸、半脸、侧脸、带手势或工作场景都可以，不需要全部都有。
```

收到图片后，助手会判断：

- 哪张适合做主图
- 哪张适合备用
- 哪些图片风险较高

然后继续问业务画像，不能停住。

### 第二轮：业务画像

用户按竖列填写：

```text
- 你是谁：
- 服务谁：
- 解决什么问题：
- 这篇内容讲什么：
- 希望用户看完做什么：
```

设计原则：

- 不要让用户写复杂商业计划。
- 只需要具体到人群和结果。
- 示例必须使用虚构身份和虚构品牌。

### 第三轮：配套素材

用户可以上传：

```text
- 产品原型 / App截图 / 网页截图
- Logo / 工具界面 / 产品图
- 课程页 / 交付物截图 / 案例截图
- 数据结果 / 用户评价 / 证明材料
- 过往封面 / 对标账号封面 / 品牌色参考
```

没有素材时，可以直接回复：

```text
无
```

## 连续生成模式

这个工具不是“一张封面一个新对话”。

在同一个聊天里，用户完成第一张封面后，可以继续说：

```text
沿用上次规范，再做一张，主题是……
```

助手需要复用：

- 个人形象图
- 业务背景
- 封面视觉规范和上一张使用的风格
- 可复用素材

如果用户没有说清楚是否沿用，助手会问：

```text
这次封面要怎么沿用上次信息？回复编号即可：
1. 沿用上次人设和封面规范，只换本次主题/素材
2. 沿用人设，但重新输入业务画像
3. 全部重新开始
```

注意：“全部重新开始”只代表当前封面重新收集素材和主题，不代表忘掉聊天里的历史上下文。后续仍然要提供：

```text
沿用上一次业务画像
沿用上一次风格
```

这里的“上一次”永远指最近完成的那一张。例如第 5 次沿用第 4 次，第 12 次沿用第 11 次。

## 最终输出结构

每张封面完成时，输出：

```text
1. 封面策略判断
2. 标题方案
3. 视觉方案
4. 小红书封面提示词
5. 出图复核清单
6. 本次封面规范沉淀
```

其中“小红书封面提示词”要能直接复制到生图工具里使用。

## 多选与 A/B 测试

当用户本应单选但选择多个编号时：

- 选 2 个：先问用户是要“两套完整提示词”，还是“一套主方案 + 一个变体”。
- 选 3 个以上：提醒用户先收敛到最想测试的 2 个。
- 如果用户明确说“都生成”“A/B测试”，直接输出两套完整方案。

两套完整方案要包含 A 方案、B 方案和 A/B 测试建议。主方案 + 变体则先输出完整主方案，再补一个差异化变体提示词。

## 风格限定机制

Skill 不应该只生成一种“大标题 + 人物 + 卡片”的封面。每次选完封面重点后，必须进入一次风格限定。

规则：

- 每次展示完整 12 个风格。
- AI 根据当前内容、素材和目标人群推荐 3 个最合适的风格。
- 用户可以按视觉形式选编号，也可以按内容用途回复，例如“产品测评”“课程教程”“个人IP”“案例结果”。
- 同一聊天内连续生成时，用户可以回复“延续风格”或“换个风格”。
- 如果用户连续生成多张都很像，助手可以提醒一次是否换风格，但不要反复打断用户。

完整风格库：

```text
1. 爆款大字压顶风
2. 巨字拆分冲击风
3. 痛点红标警示风
4. 商业顾问杂志风
5. 访谈人物海报风
6. 白板框架讲解风
7. 产品界面主视觉风
8. 工具流程信息图风
9. 深色效率工作流风
10. 教程清单风
11. 小白科普问答风
12. 案例结果证明风
```

## 封面文字与标题分工

封面文字不要只是发布标题的拆分版。

- 封面文字：用于图片大字，负责点击触发，优先 3-8 个字或 2 行短句。
- 发布标题：用于小红书笔记标题，负责解释内容方向和搜索入口，约 15-24 字。
- 两者要互补：封面制造“为什么/真的吗/这说的是我吗”，标题说明“这条内容讲什么”。

优先使用这些封面文字钩子：

```text
A 不等于 B
越 A 越 B
做了 X 却 Y
你以为 A，其实 B
只有一个关键
就这一件
真相 / 潜规则 / 没人说
```

商业类内容的人群词要尽量前置，例如老板、生意、获客、成交、合伙人、外包、客服、团队。

## 人物保真规则

这是本项目的重点规则。

生成封面提示词时，必须强调：

```text
人物必须最大程度贴近原图本人：脸型、五官比例、眼镜、发型、年龄感、气质、身材比例和服装都要接近参考图。

不要把人物美化、重绘、换脸或改成另一个人。

如果生图模型容易改脸，必要时可以 100% 保留原图人物形象，只重新设计背景、标题和辅助元素。
```

豆包或其他容易改脸的平台，建议额外加入：

```text
人物必须沿用参考图原始人物，不要重新生成另一张脸；保持原图脸型、五官、眼镜、发型、年龄感和真实气质，必要时直接保留原图人物，只替换背景、标题和卡片元素。

构图保持简单：一个主标题、一个人物、一个证明卡片、最多一个辅助小标题。不要复杂装饰，不要多层半透明效果，不要大量小字标签。人物完整清晰，不裁掉半张脸，不把脸融进背景。
```

## 生图平台建议

根据当前测试反馈：

### GPT / Codex

推荐作为优先生图路径。

优点：

- 人物相似度较高
- 封面完成度较好
- 对提示词理解较稳定

适合：

- 最终成图
- 人物 IP 封面
- 需要较高人物相似度的封面

### 豆包

可用，但需要更保守的提示词。

注意：

- 容易改人物面部特征
- 需要强制锁定人物原貌
- 构图不宜太复杂

适合：

- 备用生图
- 简洁封面
- 不需要过度复杂设计的封面

### Kimi

不建议作为最终生图主力。

更适合：

- 收集信息
- 生成标题
- 梳理封面策略
- 写生图提示词
- 复盘图片问题

如果强行用 Kimi 生图，需要提醒用户：

- 人物相似度可能不稳定
- 设计质感可能不稳定
- 要使用更简单、更保守的提示词

### Claude / DeepSeek / Hermes

适合做：

- 对话收集
- 标题策略
- 提示词生成
- 工作流执行

如果模型本身不支持生图，就只能生成提示词，不能生成最终封面图。

## 隐私与公开发布规则

这个项目准备公开发布到 GitHub 和小红书，因此必须避免暴露作者真实身份和真实业务信息。

规则：

- 不在示例中出现作者真实姓名
- 不在示例中出现作者真实业务
- 不在示例中出现真实客户、真实账号、联系方式
- 不使用私人截图或未授权案例
- 所有示例统一使用虚构人物、虚构品牌、通用行业
- 用户提供的敏感信息只用于当前封面，不要变成通用示例

当前示例采用：

- “小鹿效率笔记”：虚构知识博主
- “青禾空间整理”：虚构收纳顾问

这些示例用于教学，不指向真实个人或业务。
